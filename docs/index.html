问题：Agent 很强，但不守规矩

  • 🎲
    结果看运气 同样的任务，十次跑出八种结果
  • 🙈
    跳过该做的事 让它 review 再提交，它偏要一步到位直接 push
  • 🏃
    越做越飘 前几步还好，后面开始自由发挥、偷工减料
  • 💀
    出错就废 中间一步翻车，前面的工作全白费
根本原因：Agent 没有工作纪律。你写再长的 prompt，它该跳过的还是会跳过。

解法：把纪律写进流程，让引擎来执法

uwf 把工作流程固化成 YAML 状态机，由引擎强制执行每一步。Agent 不再有机会跳步、偷懒或自由发挥 — 它只能在当前角色的职责范围内工作，输出必须通过 Schema 验证，下一步由引擎决定。

你定义 规矩 (YAML)
→
引擎执行 状态机路由
→
Agent 只管 干活（单步）

实际场景：代码 Review

场景：让 Agent Review 一个 PR 然后合并

❌ 没有 uwf

Agent 扫了一眼 diff，说"看起来不错"，直接 approve + merge。Review 形同虚设。

✅ 使用 uwf

  • reviewer 不能跳过审查直接合并 — 它的职责里没有合并能力
  • reviewer 必须输出 approved 或 rejected + 具体理由 — Schema 强制验证
  • 只有 approved 才会路由到 merger — 引擎决定流程，不是 Agent
  • rejected 会把意见写回 PR 然后结束 — 不合格就是不能过
name: review-pr
roles:
  reviewer:
    goal: "逐文件审查代码变更"
    procedure: |
      1. 获取 PR diff
      2. 逐文件检查类型安全、错误处理、边界情况
      3. 判定：approved 或 rejected，必须给出具体理由
    frontmatter:
      oneOf:
        - properties:
            $status: { const: "approved" }
            comments: { type: string }
        - properties:
            $status: { const: "rejected" }
            comments: { type: string }

graph:
  reviewer:
    approved: { role: commenter, prompt: "写入意见并 approve" }
    rejected: { role: commenter, prompt: "写入意见，不合并" }
  commenter:
    merge:  { role: merger, prompt: "执行合并" }
    done:   { role: $END, prompt: "Review 完成，不合并" }

核心机制

🔒 角色隔离
每个角色只有自己的职责和能力。reviewer 不能 merge，developer 不能 approve。
📐 Schema 验证
输出必须符合 JSON Schema。缺字段、格式错都会被引擎拦下，不达标就重来。
🔀 状态路由
下一步走哪条路由引擎按 $status 决定，Agent 无法自作主张。
🗄️ 不可变历史
每一步存入内容寻址存储（CAS），全程可审计、可追溯、可 fork。
⏸ 暂停恢复
遇到不确定的事 Agent 挂起等人类决策，而不是瞎猜着往下跑。

对比

直接 Prompt uwf
流程遵守 ✗ 靠 Agent 自觉 ✓ 引擎强制，跳不过
输出质量 ✗ 时好时坏 ✓ Schema 兜底
可预测性 ✗ 每次不一样 ✓ 同一 workflow，同一流程
可恢复性 ✗ 翻车重来 ✓ 任意步骤 resume / fork
可审计 ✗ 聊天记录翻半天 ✓ CAS 不可变链，每步可查

适用场景

任何需要纪律的流程都适用：

solve-issue
先分析 → 再实现 → 再验证，不能跳步
review-pr
审查 → 评论 → 合并/驳回，质量关卡
release
changelog → 版本号 → 发布 → 通知，每步有检查
brainstorm
发散 → 收敛 → 评估，结构化思考不跑偏
Agent 是工人，Workflow 是工序卡。工人可以换，工序不能乱。
